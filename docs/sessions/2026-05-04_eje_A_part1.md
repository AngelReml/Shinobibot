## A1 — agentskills.io spec snapshot

Fetch directo a <https://agentskills.io/specification>. Hallazgos clave:

- Una skill = un directorio. Único archivo obligatorio: `SKILL.md` con YAML frontmatter + cuerpo Markdown libre.
- Frontmatter: `name` (1-64 kebab-case, `[a-z0-9-]+`, sin `--`, **debe coincidir con nombre del directorio**) y `description` (1-1024 chars). Opcionales: `license`, `compatibility` (≤500), `metadata` (map string→string), `allowed-tools` (experimental).
- **No hay campo de permisos formal** ni de triggers. La activación es description-driven por LLM.
- **No declara engine**. `scripts/` es path libre; el cliente decide cómo ejecutar.
- Carga progresiva: discovery (~100 toks) → activation (≤5000 toks recomendado) → execution (refs/scripts on-demand).
- Body recomendado <500 líneas; trozos extensos en `references/`.

Documento detallado en `docs/research/agentskills_diff.md` con tabla de mapeo (Shinobi v1.0 → v1.1) y decisión de qué metadata Shinobi-específica vive en `metadata.shinobi.*`.

## Decisiones autónomas

1. **Mantener `.mjs` interno + emitir bundle** — no reescribimos el runtime de skills (registerTool sigue siendo la fuente de verdad); `agentskills.io` es una **vista de exportación** que también acepta import. Esto evita migración masiva del SkillsAgent y SkillGenerator existentes.
2. **`metadata.shinobi.*` para round-trip lossless** — todos los campos Shinobi (id, status, visibility, schema_version, legacy_name, engine) viven aquí. Otros agentes ignoran lo que no entienden; Shinobi reconstruye un `SkillEntry` exacto.
3. **`allowed-tools` emitido pero no aplicado** — el spec lo marca experimental; lo serializamos por compatibilidad pero el control real sigue en el cliente.
4. **Script de migración en `shinobibot/scripts/`** (no en OG) — porque toca `%APPDATA%/Shinobi/`, que es path Shinobi-específico, no datos de OpenGravity.
