## C4 — Dashboard live

Decisión técnica documentada en commit: **SSE en lugar de WebSocket**. Razón:
- Canal read-only (server → client)
- `EventSource` nativo en navegador, sin dependencia `ws` extra
- Reconexión automática gratis

Archivos:
- `src/benchmark/shinobi/events.ts` — bus `EventEmitter` único, fan-out a clientes SSE
- `runner.ts` y `improve.ts` emiten 8 tipos de evento: `run_started`, `task_start`, `task_done`, `improve_started`, `candidate_generated`, `skill_approved`, `gap_resolved`, `ledger_appended`, `run_completed`
- `gateway.ts` monta `GET /v1/benchmark/stream` ANTES del middleware de auth para que `EventSource` pueda autenticar vía `?key=` (no permite headers custom)
- `gateway.ts` también sirve `/dashboard/live` → `dashboard/improve_live.html`
- Dashboard standalone con 3 paneles: 30-task grid, scoreboard + event stream, auto-skills + chain stream. Tema dark monoespaciado (paleta tipo Linear/Vercel).

Test E2E (`__tests__/sse.test.ts`): levanta gateway en subprocess, abre EventSource, lanza run con 2 tareas y verifica que llegan los 5 tipos esperados de eventos. **PASS, 8 eventos.**
