## G3 — Release CI

`.github/workflows/release.yml` — fires on push to main que toca `package.json` o el workflow mismo. También dispatch manual con `version` opcional.

- Detecta versión (campo `version` de package.json o input manual)
- Si el tag ya existe → skip (idempotente).
- `npm ci` + `node build_sea.mjs` (esbuild → `build/shinobi.exe`)
- Stage en `release-artefacts/`: `Shinobi-<ver>.exe`, `shinobi-portable-<ver>.zip`, `ShinobiSetup-<ver>.exe` (opcional, B1).
- `node scripts/release/generate_changelog.mjs > CHANGELOG.md` agrupa commits por tipo (feat/fix/perf/refactor/test/docs/build/ci/release/chore/style/other).
- Push tag + `softprops/action-gh-release@v2` con assets.
- Notificación Discord opcional (`DISCORD_WEBHOOK_URL` secret) — silent skip si no está.

`scripts/release/generate_changelog.mjs` testeado localmente: 127 commits desde el último tag agrupados correctamente por type.

## G1 — Issue triage

`scripts/triage/classify_issue.mjs`:
- Lee JSON `{title, body}` por stdin.
- Si `TRIAGE_LLM_KEY` está set → llama OpenRouter compat (modelo `gpt-4o-mini` por defecto, 8 tokens max). Mismo cliente sirve OpenAI/Anthropic mediante override de `TRIAGE_LLM_BASE_URL` y `TRIAGE_LLM_MODEL`.
- Si la LLM falla o no hay key → **clasificador heurístico** por keywords (bug/feature/question/docs/other).
- Detecta lenguaje ES/EN por keywords y devuelve un `suggested_reply` bilingüe.
- Output: `{category, label, suggested_reply, classifier, detected_language}`.

`.github/workflows/issue_triage.yml`:
- Trigger `issues.opened`.
- Pipe del payload al classifier, captura output, aplica label vía `gh issue edit`, postea reply con `gh issue comment`.

Smoke test local con 3 issues sintéticas:
- "Crash on startup..." → `bug` (EN)
- "Add support for Linux" → `feature` (EN)
- "Cómo configuro la API key?" → `question` (ES)

3/3 correctos.

## G2 — Telemetría opt-in

### Cliente (`Shinobibot/src/telemetry/telemetry.ts`)

- Estado en `%APPDATA%/Shinobi/telemetry.json` (resuelto lazy):
  ```json
  { "opted_in": false, "anonymous_id": "<uuid>", "endpoint": "...", "asked_at": "..." }
  ```
- API: `loadConfig`, `saveConfig`, `setOptIn`, `ensureConfigInitialized`, `emit(event, properties)`, `summary()`.
- `emit()` rechaza eventos fuera del allowlist (`session_start`, `command`, `error`, `skill_loaded`, `demo_run`, `update_check`, `session_end`).
- `sanitizeProperties` clamp strings a 64 chars, sólo string/number/boolean.
- Default opt-out: `emit()` retorna `{sent: false, reason: 'opted out'}` si `opted_in=false`.
- Timeout 3s, falla silenciosa.

### Wizard (`first_run_wizard.ts`)

Tras la pregunta de memory_path, añadida una pregunta:

```
Anonymous telemetry (no prompts/paths, just aggregate counters)? [y/N]:
```

Default = NO. La respuesta se persiste en `telemetry.json` durante onboarding.

### CLI

`shinobi telemetry [on|off|status]`:
- `on` → setOptIn(true), imprime UUID truncado.
- `off` → setOptIn(false).
- `status` → JSON con config + endpoint resuelto.

### Backend (`OpenGravity`)

`src/telemetry/store.ts`:
- Append-only `datasets_v2/telemetry/events.jsonl`.
- `aggregate()` produce: `total_events`, `unique_installs`, `by_event/platform/version`, `active_24h/7d`, `first_seen/last_seen`.

`src/api/v1/telemetry_router.ts`:
- `POST /v1/telemetry/event` — valida campos, allowlist, rate limit 60/min por anonymous_id (in-memory bucket).
- `GET /v1/telemetry/summary` — agregado.
- `GET /v1/telemetry/recent?limit=N` — tail.
- Montado bajo el auth middleware (cliente envía `X-Shinobi-Key`).

`dashboard/telemetry.html` — tema dark monoespaciado, prompt para X-Shinobi-Key, fetch al summary + recent. Servido en `GET /dashboard/telemetry` desde `gateway.ts`.
