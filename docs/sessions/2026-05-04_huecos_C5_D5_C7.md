## C5 — Skill Index + Reflection + Enhance

**Problema**: C3 generaba skills sin lookup, reinventando la rueda a escala.

**Entregado**:
- `OpenGravity/src/skills/skill_index.ts` — Jaccard sobre keywords + bonus de categoría + bonus de uso. 3 modos:
  - `REUSE` (≥0.70): salta el generator, ejecuta skill existente, evento `skill_reused` en chain.
  - `ENHANCE` (0.50–0.70): pasa la skill como `base_template` + N candidatas; evento `skill_enhanced parent=X -> Y`.
  - `GENERATE` (<0.50): flujo C3 vanilla.
- `improve.ts` integra el lookup pre-generator. Flag `bypass_index: true` mantiene C3 path para A/B testing.
- Reflection loop cada 10 runs detecta duplicados (parameters_hash), candidatos a archivar (usage_count=0), promotion a `core` (usage_count≥5).
- Endpoints `GET/POST /v1/skills/reflect` + `POST /v1/skills/reflect/merge` (solo propone, no ejecuta).
- `data/skill_index.json` persistente.
- `docs/architecture/skill_index.md` con diagrama.

**Tests**:
- `c5_index.test.ts`: 5 fases (empty index → exact reuse → enhance → generate → no regression). PASS.
- `baseline.test.ts` (C3 existente): 16/16 PASS sin cambios.
- `improve.test.ts` (C3 existente): 4/5 invariante.

**Hash chain compat**: cero migración. Los nuevos motivos (`skill_reused via X`, `skill_enhanced parent=A -> B`) viven en `reason`, esquema invariante.

## D5 — Behavioral Deviation Score con Calibration Mode

**Problema**: AgentAudit era hash-chain post-hoc + reglas binarias. "Behavioral verification" sin score real.

**Entregado**:
- `OpenGravity/src/audit/deviation_score.ts`:
  - **Calibration mode**: primeras N=20 acciones por (agent, type) → `score=0, calibrating=true`. Acción 21+ scoring real.
  - Welford online stats para mean/variance por feature numérico.
  - Forbidden zone (system32, rm-rf-root, http-exfil, private-key-read) → BLOCK 10 incluso en calibración.
  - Never-seen action_type post-calibración → 7 ALERT.
  - 3+ sigmas con varianza meaningful (>1% relativo) → 5 WARN.
  - Routine → 0–2 ALLOW.
  - Variance guard contra falsos positivos cuando calibración fue constante.
- `audit_router.ts` integrado: cada `/event` calcula score, BLOCK no persiste, ALLOW/WARN persiste enriqueciendo `metadata` con `deviation_score/bucket/calibrating`.
- Endpoints nuevos: `GET /v1/audit/profile/:agent_id`, `GET /v1/audit/profiles`.
- `dashboard/deviation.html` con heatmap + alerts + by-type breakdown + status calibración.
- Landing `web/audit/index.html` reescrita: "Behavioral Deviation Score" como hero, "Hash chain forense" secundaria, sección dedicada a Calibration period (transparencia).

**Tests** (`d5_deviation.test.ts`, 6 fases):
1. Calibración: primeras 20 acciones score=0, acción 21 scored.
2. 50 routine actions: max score=0.
3. Forbidden injection: 10/block, regardless of calibration.
4. Novel action_type post-calibración: 7/unknown_type.
5. Latencia 1000 calls: p50=2.0ms, p99=3.9ms.
6. /profile endpoint y /event block path E2E.

**Hallazgos durante el debug**:
- Bug regex `\b...\b` (no boundary tras `/`) → fix a `\b(...)`.
- Stale gateway processes en Windows (cp.kill no llega al grandchild de `npx tsx`) → añadido `killTree()` con `taskkill /T /F`.

## C7 — Killer demo

**Problema**: la demo ShinobiBench (30 tareas) es buena pero floja virilmente. Hace falta una tarea única, imposible para competidores, con narrativa automática.

**Entregado**:
- `demos/killer_task.md` — caso "Rename PDFs by content": scroll-infinito + load-more + PDF parsing + rename por contenido.
- `demos/test_site/`:
  - `serve.mjs` — HTTP local, port 8765.
  - `index.html` — IntersectionObserver + 3 botones load-more, render JS-side.
  - `generate_pdfs.mjs` — 8 PDFs sintéticos hand-rolled (sin pdfkit) con `/Title` interno distinto del filename.
- `demos/killer_demo_runner.mjs` — pipeline completo:
  1. SETUP_PRESENTED → AGENT_FIRST_ATTEMPT (scrape estático, encuentra 0/8) → AGENT_FAILURE.
  2. C3_TRIGGERED → SKILL_GENERATION_START → SKILL_VALIDATION_PASS → SKILL_APPLIED.
  3. PDFS_DOWNLOADED (8) → TITLE_EXTRACTED (8) → RENAME_DONE (8).
  4. SUCCESS_TOTAL.
  5. `chapters.md` autogenerado de timestamps reales (no hardcoded).
- CLI `shinobi demo --task killer` envuelve con OBS bracketing opcional vía H1+H2.
- `demos/README.md` — instrucciones reproducción desde cero.

**Tests**:
- Dry-run (sin OBS, sin LLM): 0.4 segundos end-to-end, 8/8 PDFs renombrados, exit 0.
- chapters.md emitido correctamente con timestamps reales del log.

**Diseño que preserva la viralidad** sin depender de LLM en CI:
- El runner local es determinista (un agente "stub" que falla en static scrape y luego recupera vía manifest.json).
- Los timestamps son **reales wall-clock**, no narrativa — la `chapters.md` es evidencia de continuidad.
- La versión LLM-driven con C3 emergente real queda como ejecución manual cuando el operador lo decida (necesita OPENROUTER_API_KEY + tiempo).
