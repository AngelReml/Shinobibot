## Reglas operativas honradas

- **Backup obligatorio antes de sobrescribir**: `<file>.backup-<UTC>` o `<dir>.backup-<UTC>`. Si la copia falla, se aborta el paso y se reporta en `errors[]`.
- **Default `--dry-run`**: si el usuario olvida el flag, no pasa nada.
- **`.env` no se sobrescribe ciegamente**: se preservan claves existentes y sólo se añaden las nuevas bajo un comentario `# imported from hermes <UTC>`.
- **Variables no-API-key se ignoran** al parsear `.env` Hermes (verificado en test: `OTHER_VAR` no aparece en el `.env` resultante).

## Decisiones autónomas

1. **YAML parser ad-hoc** en vez de añadir dependencia (`js-yaml`). Hermes config es plano (`key: value` top-level), suficiente.
2. **Memory por heading**: `## Build pipeline` se almacena como un `MemoryEntry` independiente. Esto da grano fino para recall semántico. Si Hermes no usa headings, todo el archivo es un único entry.
3. **Lazy import de `MemoryStore`**: `better-sqlite3` es nativo y pesado; no se carga durante `--dry-run`. El test es resiliente a fallos al cargarlo.
4. **No tocamos el directorio Hermes original**: sólo lectura. Si el usuario pierde Shinobi puede re-importar.
5. **Cleanup de tests resiliente** a EPERM en Windows (WAL handle de SQLite). Tmp queda colgando ocasionalmente; aceptable para tests.
