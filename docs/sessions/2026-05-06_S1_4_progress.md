## CHECKPOINT 4 — Baseline (BLOCKED)

**Intentos de captura**: 2.

**Intento 1** (s1_4_runner.ts inicial):
- 11/15 runs OK, 4 fallos (T2/T3/T4/T5 todos en run 3) por 429 OpenAI.
- **Bug detectado en el runner**: cleanup de cache F1 buscaba archivos por nombre `digininja|DVWA` pero los archivos se nombran solo con SHA → cache nunca se limpiaba → T1 reusó cache de un run F2 anterior dando verdicts inconsistentes (run1 FAIL/high, run2/3 PASS/low). Datos contaminados, no válidos para baseline.
- Fix aplicado al runner: cleanup ahora usa `DVWA_SHA` constante.

**Intento 2** (con fix del cleanup + sleeps de 25s entre tareas):
- T2.run1 OK.
- T3.run1 falla con 429 sostenido. T5.run1 falla. T4.run1 falla. T1.run1 falla en final synth con 429.
- 1/15 runs OK antes del crash. Carpeta `baseline/` vacía.

**Diagnóstico del blocker**:
- La API OpenAI (gpt-4o + gpt-4o-mini) está rate-limited con 429 sostenido.
- En tier 1 estándar de OpenAI, gpt-4o tiene ~30k TPM. Un solo audit DVWA con voting=3 + code_reviewer consume ~50k tokens. Cualquier corrida agresiva golpea el límite.
- Sleeps de 25s entre tareas no son suficientes para reset del bucket TPM.
- Este es exactamente el blocker del **R3** del briefing: *"Una tarea de evaluación requiere acceso a un servicio externo que está caído"*. El servicio no está caído pero está aplicando back-pressure equivalente.

**Por qué NO improviso** (R3 explícito):
- NO bajo voting=3 a voting=1 — rompería F1 y el gate, datos de baseline no comparables.
- NO cambio gpt-4o → gpt-4o-mini para synth — cambiaría el sistema bajo medida, datos no comparables.
- NO reduzco runs de 3 a 1 — perdería medición de varianza, gate del CHECKPOINT 4 dice "3 corridas independientes".

**Rutas disponibles para desbloquear** (decisión humana):

1. **Esperar 1-2 horas** antes de re-ejecutar `npx tsx scripts/s1_4_runner.ts baseline`. El bucket TPM se resetea con el tiempo. Más sleeps en el runner (60-90s) ayudan pero alargan el wall-clock significativamente.
2. **Subir tier OpenAI** (requiere pago) — saldría de tier 1 a tier 2/3 con 200k+ TPM. Resuelve definitivamente.
3. **Cambiar a Ruta B (`glm-4.7-flash` vía OpenRouter)** — el plan original lo proponía. Requiere configurar `OPENROUTER_API_KEY` en `.env`. Las cuotas de OpenRouter son distintas y mucho más laxas para modelos no-Anthropic. Si esta es la ruta elegida, cambia la configuración en §0 del CHECKPOINT 3 y se re-ejecuta baseline desde cero.

**Lo que SÍ está committeado en este checkpoint**:
- `scripts/s1_4_runner.ts` con el fix del cleanup F1 y sleeps de 25s entre tareas. Listo para re-ejecutar cuando se desbloquee.
- Carpeta `docs/s1_4/baseline/` vacía (esperando outputs).

## Hallazgos colaterales

**Bug en runner inicial**: la función `t1()` de `scripts/s1_4_runner.ts` limpiaba archivos del cache F1 buscando `/digininja|DVWA/i.test(f)` en el nombre, pero los archivos del cache se nombran con SHA puro (e.g. `33e364c5..._committee.json`). El cache nunca se limpiaba → audits sucesivos del mismo SHA reusaban report de runs anteriores. **Fixed**: ahora usa `DVWA_SHA` constante y elimina archivos `<SHA>_*` directamente.

**Variabilidad observada (intento 1)**: T1.run1 (Reader fresh + Committee fresh) → FAIL/high. T1.run2 y T1.run3 (Reader cache hit + Committee fresh) → PASS/low. Esto sugiere que el committee con voting=3 y temp=0 sobre el MISMO report cacheado puede dar verdicts distintos. Es una observación interesante pero no concluyente porque los runs 2/3 cargaron un report cacheado de la corrida F2 (no del intento actual). Para diagnóstico real se necesita el baseline desbloqueado.
