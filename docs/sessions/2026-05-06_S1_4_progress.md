## Hallazgos colaterales

**Bug en runner inicial**: la función `t1()` de `scripts/s1_4_runner.ts` limpiaba archivos del cache F1 buscando `/digininja|DVWA/i.test(f)` en el nombre, pero los archivos del cache se nombran con SHA puro (e.g. `33e364c5..._committee.json`). El cache nunca se limpiaba → audits sucesivos del mismo SHA reusaban report de runs anteriores. **Fixed**: ahora usa `DVWA_SHA` constante y elimina archivos `<SHA>_*` directamente.

**Variabilidad observada (intento 1)**: T1.run1 (Reader fresh + Committee fresh) → FAIL/high. T1.run2 y T1.run3 (Reader cache hit + Committee fresh) → PASS/low. Esto sugiere que el committee con voting=3 y temp=0 sobre el MISMO report cacheado puede dar verdicts distintos. Es una observación interesante pero no concluyente porque los runs 2/3 cargaron un report cacheado de la corrida F2 (no del intento actual). Para diagnóstico real se necesita el baseline desbloqueado.

### Hallazgos colaterales con impacto retroactivo

**Cleanup del cache F1 buscaba por nombre cuando los archivos se guardan por SHA** (commits `01c2e35` original y `<commit retro>` documentación). El bug del runner es un síntoma de un patrón que aplica a **cualquier** consumidor del cache F1 que quiera invalidarlo: el filename en `audits/.machine/` es `<SHA>_<suffix>.json` (sin owner ni repo), por lo que filtrar por `digininja|DVWA` o cualquier substring no-SHA NUNCA encuentra los archivos. Cualquier audit posterior a F1 sobre el mismo SHA leerá del cache silenciosamente — eso es el comportamiento DESEADO del cache (reproducibilidad por SHA fijo) pero es una sorpresa para tests/baselines que esperan regenerar. La forma correcta de invalidar es: borrar `<SHA>_*` directamente, o documentar que "re-auditar el mismo SHA siempre devuelve el report cacheado salvo que se borre". Este patrón debe respetarse en cualquier futuro código que orqueste batches de audits sobre el mismo SHA (ej. CI, regression tests post-deploy).
