## A3 — Cliente MCP v2

**Spec consultado**: <https://modelcontextprotocol.io/specification/2025-06-18>.

### Carencias del cliente v1 (mcp_client.ts) — top 5

1. No envía `notifications/initialized` tras `initialize`. Spec MUST.
2. `protocolVersion: 2024-11-05` desactualizada (actual: `2025-06-18`).
3. SSE stub sin implementar; spec actual prefiere "Streamable HTTP".
4. Sin `sampling/createMessage` (server → cliente).
5. Sin tool filtering, sin resources/prompts discovery.

(Lista completa de 12 gaps en `docs/research/mcp_audit.md`.)

### Cliente v2 entregado

- `OpenGravity/src/mcp/mcp_transports.ts` — `StdioTransport` (stdin/stdout JSON-line) + `StreamableHttpTransport` (POST + SSE-like streaming + `MCP-Protocol-Version` header).
- `OpenGravity/src/mcp/mcp_client_v2.ts` — `MCPClientV2`:
  - Lifecycle correcto: initialize → wait response → `notifications/initialized` → operate
  - Capabilities declaradas: `roots`, `sampling`, `elicitation`
  - Tool filtering vía `allow_tools` / `deny_tools` (exact, glob `*`, regex `/.../`)
  - Sampling callback (`onSamplingRequest`) — el host de OG puede delegar al `OpenRouterClient` existente
  - `discover_resources` / `discover_prompts` opt-in
  - Shutdown ordenado: stdin close → 1.5s espera → SIGTERM → 0.5s → SIGKILL
  - `MCPError` con `code` + `data` para diagnóstico
  - Race-safe `safeSend` para mensajes que llegan durante shutdown

V1 (`mcp_client.ts`) intacto — `capability_installation_pipeline.ts` sigue funcionando. Migración gradual.

## A5 — Hermes upstream watcher

`src/watchers/hermes_watcher.ts`:

- Polling default cada **24h**, pero hace un primer check 5s después de start.
- Estado persistido en `%APPDATA%/Shinobi/hermes_watcher.json`:
  - `last_seen_tag`, `last_check_iso`, `pending_notice`
- API:
  - `checkOnce(opts)` — single check, returns notice if new tag
  - `popPendingNotice()` — consume + clear
  - `renderNotice(notice)` — formato CLI (caja unicode con tag, fecha, assets, sugerencia `shinobi import hermes`)
  - `startWatcher(opts)` — `setInterval` con `unref()` para no bloquear el process
- Repo target: `NousResearch/hermes-agent` (override con `HERMES_WATCH_REPO` env).
- Drafts y prereleases se ignoran.
- Si la GitHub API falla, marca `last_check_iso` y propaga el error.

CLI: en `scripts/shinobi.ts` después de `checkKernel()`, intenta cargar el watcher, muestra el notice pendiente (si hay) y arranca el watcher en background.
