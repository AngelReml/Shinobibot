## B2 — Auto-update

### Backend (`OpenGravity`)

Endpoint público (no auth) `GET /v1/version`. Lee `SHINOBI_LATEST_*` env vars:

```json
{
  "success": true,
  "component": "shinobi",
  "latest_version": "1.0.0",
  "download_url": "...",
  "sha256": "...",
  "released_at": "...",
  "channel": "stable",
  "notes_url": "..."
}
```

Default values son seguros (latest=1.0.0, no download): aunque la release pipeline no esté lista, el endpoint devuelve 200 sin ofrecer update.

### Cliente (`Shinobibot`)

`src/updater/version_check.ts`:
- `fetchManifest()`, `checkForUpdate()`, `declineVersion()`, `renderOffer()`
- State file en `%APPDATA%/Shinobi/updater_state.json` (resuelto **lazy** para que tests overrideen `APPDATA`).
- compareSemver simple (major.minor.patch).
- Si la kernel está caída → `null`, sin error visible al usuario.

`src/updater/install_update.ts`:
- `runUpdate(offer)` y `fetchAndInstall(opts)`.
- Descarga al `%TEMP%/shinobi-setup-<ver>.exe` con `fetch` global.
- Verifica `sha256` si la manifest lo provee. Mismatch → abort.
- `--dry-run` se queda en download + verify.
- Modo real: `spawn(exe, ['/SILENT','/NORESTART'], { detached:true, stdio:'ignore' })` + `unref()` + el process actual exits en 2s.

### CLI

- Startup non-blocking: banner si hay update.
- `shinobi update --check` solo muestra el offer.
- `shinobi update --dry-run` descarga + sha256.
- `shinobi update` ejecuta el flujo completo.

### Test E2E

`src/updater/__tests__/update_e2e.test.ts`:
1. Stub kernel HTTP en port random sirve `/v1/version` y `/install/shinobi-setup.exe`.
2. `checkForUpdate` con `latest=999.0.0` → offer.
3. `declineVersion(999.0.0)` → siguiente check null.
4. `fetchAndInstall(--dry-run)` → descarga + sha256 verificado.

**`[b2-e2e] OK`**.
