+## Contrato de retry/backoff para `hermes_watcher.ts`
+
+Antes de tocar el código, este es el contrato que debe cumplir el watcher de Hermes ante fallos transitorios (timeouts, 5xx, disconnects):
+
+- **Política**: reintentos con backoff exponencial + jitter.
+- **Intentos máximos**: 5 por evento (el primer intento + 4 reintentos).
+- **Delay base**: 500 ms.
+- **Fórmula**: `delay_n = min(base * 2^n, cap) + random(0, 250ms)` donde `n` es el número de reintento (0-indexed) y `cap = 30_000 ms`.
+- **Errores reintetables**: `ETIMEDOUT`, `ECONNRESET`, `ECONNREFUSED`, `EAI_AGAIN`, HTTP 408/425/429/5xx.
+- **Errores NO reintetables**: 4xx distintos de 408/425/429 (se propagan inmediatamente).
+- **Respeto a `Retry-After`**: si la respuesta trae `Retry-After` (segundos o fecha), se usa ese valor en lugar de la fórmula, acotado por `cap`.
+- **Cancelación**: el loop debe abortar si el `AbortSignal` del watcher se dispara, incluso durmiendo en el backoff.
+- **Observabilidad**: cada reintento emite un log estructurado `{ event, attempt, delay_ms, reason }` y se incrementa la métrica `hermes_watcher_retries_total{reason}`.
+- **Dead-letter**: al agotar los 5 intentos, el evento se envía a la cola `hermes:dead` con el último error serializado, y se emite `hermes_watcher_dead_total`.
+
+Cualquier PR que modifique `hermes_watcher.ts` debe incluir tests que cubran: éxito al primer intento, éxito tras N reintentos, error no reintetable, respeto a `Retry-After`, cancelación por `AbortSignal` y dead-letter al agotar intentos.

+# runner.ts hardening spec
+
+`src/utils/runner.ts` currently shells out via `execSync(commandString)`.
+LLM-generated strings reach this call directly, which means any prompt
+injection that produces e.g. `foo & rmdir /s /q C:\\` is a shell command.
+
+## Required contract for the refactor
+
+1. **Array arguments only.** The public API MUST accept `(cmd: string,
+   args: string[], opts?)`. No single-string overload, no template-literal
+   helper — those re-introduce shell parsing.
+2. **execFile, not exec.** Use `child_process.execFile` (or `spawn` with
+   `shell: false`). This guarantees no `/bin/sh` or `cmd.exe` interpolation.
+3. **Timeout.** Default `timeout: 30_000` ms, killable via `AbortSignal`.
+   On timeout, send SIGTERM then SIGKILL after 2s grace.
+4. **Allowlist.** The `cmd` value MUST be matched against a static
+   allowlist of executable basenames (e.g. `git`, `node`, `npm`, `pwsh`)
+   before invocation. Absolute paths must be resolved and re-checked.
+5. **Output caps.** Cap captured stdout/stderr at 1 MiB each; truncate with
+   a marker rather than buffering unbounded.
+6. **No `shell: true`.** Ever. Reviewers should reject any PR that adds it.
+
+## Migration
+
+Callers that construct shell pipelines (`a | b > c`) must be rewritten to
+compose multiple `runner()` calls in JS, or to use a dedicated, reviewed
+helper that explicitly opts into shell semantics with a comment justifying
+the risk.
```

+# Dependency pinning policy
+
+## Why
+
+Without a committed lockfile, `npm install` resolves semver ranges at
+install time. A compromised minor or patch release of any transitive
+dependency would be silently pulled into the agent's runtime, which has
+filesystem write, shell execution, and desktop automation privileges.
+
+## Policy
+
+1. `package-lock.json` MUST be committed to the repository.
+2. CI MUST run `npm ci` (not `npm install`), which fails if
+   `package-lock.json` is out of sync with `package.json`.
+3. Dependency updates go through a dedicated PR that shows the lockfile
+   diff. Reviewers should skim added/changed transitive packages for
+   unexpected maintainers or scope changes.
+4. `npm audit --production` runs in CI and blocks on `high` or `critical`
+   advisories. `moderate` advisories are logged but non-blocking.
+5. Production builds (including `build_sea.mjs`) MUST be produced from a
+   clean `npm ci` install, never from a developer's mutated `node_modules`.
+
+## Exceptions
+
+None. If a dependency cannot be pinned (e.g. a git URL), it must be
+vendored into the repo and reviewed as source.
```
