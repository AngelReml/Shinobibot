```diff
+# Hermes Watcher — rate-limit & conditional-request plan
+
+`src/watchers/hermes_watcher.ts` polls GitHub on an interval derived from
+config. Today it has no awareness of the unauthenticated 60-req/hr cap,
+so a config change can silently breach the limit and the watcher will
+fail without signal.
+
+## Required behavior
+
+1. **Auth token (optional).** If `GITHUB_TOKEN` is set, send it as a
+   bearer token. This raises the quota from 60/hr to 5000/hr.
+2. **Quota logging.** On every response, log
+   `x-ratelimit-remaining` and `x-ratelimit-reset` as structured fields.
+3. **Backoff threshold.** When `remaining` drops below 10% of the cap,
+   the watcher must skip its next tick and sleep until `reset`.
+4. **Conditional requests.** Persist the `ETag` (or `Last-Modified`)
+   from each response and send it back as `If-None-Match` (or
+   `If-Modified-Since`) on the next request. 304 responses do **not**
+   count against the rate limit.
+5. **429 / secondary limits.** On HTTP 429 or a `retry-after` header,
+   honor the delay and emit a warning-level log.
+
+## Non-goals
+
+- Switching to webhooks. Polling is retained to keep the inbound
+  attack surface minimal.
+- Per-mission quota accounting.
+
+## Test hooks
+
+The watcher should accept an injected `fetch`-like function so tests
+can simulate 304, 429, and low-remaining responses without a real
+GitHub round-trip.
```

```diff
+# runner.ts — timeout policy
+
+`src/utils/runner.ts` wraps `child_process.execSync`. Today it does not
+pass a `timeout` option, so any caller — including watchers and audit
+runners that shell out to LLM-driven commands — can hang indefinitely.
+
+## Contract
+
+The runner exposes a single `run(cmd, opts)` entry point. `opts.timeout`
+(milliseconds) is **required** at the type level, with the following
+rules:
+
+- Default (when callers pass `undefined` explicitly): **30_000 ms**.
+- Hard upper bound enforced inside the runner: **300_000 ms** (5 min).
+  Values above the cap are clamped and a warning is logged.
+- On timeout, `execSync` raises; the runner must re-throw with a
+  structured error containing `{ cmd, timeoutMs, killed: true }`.
+
+## Caller obligations
+
+- `src/watchers/hermes_watcher.ts`: pass a timeout shorter than its
+  polling interval so a hung call cannot overlap the next tick.
+- `scripts/a4_run.ts` and other audit runners: pass an explicit bound
+  derived from the audit budget; do not rely on the default.
+- Any caller invoking LLM-generated command strings: must pass an
+  explicit timeout and must not exceed the hard cap.
+
+## Migration
+
+1. Add the `timeout` parameter with the default above.
+2. Add a test that asserts a sleeping command is killed at the bound.
+3. Update each call site to pass an explicit value; leave the default
+   only for interactive/dev paths.
```

```diff
+# a4_run.ts — error handling requirements
+
+`scripts/a4_run.ts` currently contains empty `catch` blocks that
+silently swallow read failures. This undermines the self-analysis
+purpose of the runner: audit-time faults, permission denials, and
+potential injection attempts become invisible.
+
+## Required changes
+
+1. **No empty catches.** Every `catch` must either:
+   - Re-throw after annotating context, or
+   - Write a structured entry to the audit log stream.
+2. **Structured log entry.** Minimum fields:
+   - `level`: `"error"` or `"warn"`.
+   - `phase`: which audit step was running.
+   - `path`: the file or directory being processed, if applicable.
+   - `errorType`: `err.name` or `err.code`.
+   - `message`: `err.message`.
+   - `llmInput`: the LLM-produced string that led here, when available
+     (redact secrets before logging).
+3. **Exit codes.** An audit pass with any `level: "error"` entries must
+   exit non-zero so CI surfaces the failure.
+
+## Rationale
+
+The committee (architect + security_auditor, consensus) flagged the
+empty catches as a high-risk gap: they mask both operational bugs and
+injection patterns in LLM-generated inputs. Routing errors through the
+same audit log the runner already produces keeps observability in one
+place.
```
